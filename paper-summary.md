# Summary: *Towards a Science of AI Agent Reliability* (Rabanser et al., arXiv:2602.16666 v2, Feb 2026, pp. 1-21)

The paper's core argument is that single-number accuracy hides what matters once an agent leaves the benchmark. Two agents with identical pass@1 can fail on a fixed identifiable subset of tasks (debuggable, deployable behind guardrails) or fail unpredictably on a different subset every run (not). Mean accuracy cannot tell you which one you have. The authors' fix is to import the multi-dimensional reliability framing from safety-critical engineering (aviation, nuclear, automotive, rail) and propose a concrete, computable analogue for AI agents: four dimensions (consistency, robustness, predictability, safety), twelve metrics, all bounded in `[0, 1]`, reported as a profile rather than collapsed.

The four dimensions are pitched cleanly. **Consistency** asks: does the agent behave the same way under the same conditions? **Robustness**: does it degrade gracefully when conditions deviate (perturbed prompts, faulty tools, renamed APIs)? **Predictability**: can the agent's confidence be trusted as a signal of when it is about to fail? **Safety**: when failures occur, how severe are the consequences? Each dimension decomposes into 2-4 sub-metrics. The metric I find most concretely useful is outcome consistency `C_out`, which normalises the per-task variance of `K = 5` repeated runs by the maximum possible Bernoulli variance `p(1-p)`. So a model with 50% accuracy and 50% per-task variance scores zero, while a model with 50% accuracy that always splits the same 250/500 tasks scores one. Normalising by `p(1-p)` decouples reliability from accuracy: a 50%-accurate model can score 0 (random failure) or 1 (deterministic on a fixed split). Safety is reported separately rather than averaged in, on the explicit ground that tail risks should not be smoothed by aggregation, which is right.

Empirically, they evaluate 14 models from OpenAI, Google, and Anthropic, spanning early 2024 to late 2025, on GAIA (165 general-assistant tasks) and a 26-task cleaned subset of τ-bench (customer-service simulation). For each model-task pair they run `K = 5` trials with temperature zero, `J = 5` prompt paraphrases, fault injection at probability 0.2, and structured environment perturbations. Headline finding (Figure 1): accuracy improves at roughly 0.21/yr on both benchmarks, while reliability improves at 0.03 to 0.10/yr. After 18 months of capability progress, the reliability gap has barely closed. Even frontier models post outcome-consistency scores in the 0.6 to 0.7 range; the same model that looks strong on the leaderboard often fails to solve the same task twice in a row.

The single nugget I think most useful for anyone designing trajectory-level metrics on coding agents is that across runs of the same model on the same task, the distributional trajectory consistency `C_traj^d` (Jensen-Shannon over action-type histograms) is much higher than the sequence-level `C_traj^s` (normalised Levenshtein over action sequences). The model picks the same kinds of actions every time but reorders them: same actions, different order. For trajectory-quality work this is the difference between scoring the agent on what it did and on the order in which it did it. If you score the latter, you reward stylistic uniformity over substantive agreement; if you score the former, you can confuse two agents that produce identical action histograms but very different patches.

Two pushbacks. First, `K = 5` at temperature zero measures sub-sampling variance: kernel non-determinism, batch-size effects, ordering of attention reductions. Reasoning models cannot have temperature configured anyway, so for them it is closer to genuine model variance, but for the others the consistency numbers are mostly a story about inference-stack variability rather than model behaviour. A consistency claim that a deployment team would actually act on probably wants temperature varied. The paper acknowledges the ambiguity but does not separate the two sources. Second, the paper's own §6 flags single-scaffold evaluation as a limitation, and on coding agents this is the gap that matters most. The same Claude 4.5 Opus running mini-SWE-agent v2, Claude Code, and Codex would post very different consistency, robustness, and resource profiles. I see this empirically in the Task 2 work in this repo, where five frontier models on the *same* scaffold span 2.3x in median turn count and 10x in cost per resolved task at near-equal pass@1. Reliability profiles measured under one scaffold should not be expected to transfer.

A connection to my own work in this repo. The paper's predictability dimension is about the agent's expressed confidence (calibration, AUROC of self-reported scores). My Task 2 finding is that an implicit predictability signal sits in the trajectory shape and the patch content themselves: `S_m(k) = P(resolved | T >= k)` collapses from 77% to 9% by turn 75 for Claude 4.5 Opus, while Gemini 3 Flash holds 76% -> 68% on the same domain. A logistic regression on trajectory shape plus patch-vs-ground-truth cosine similarity (mean-pooling Qwen2.5-Coder-1.5B over each diff on a local 4090) predicts the resolved flag at mean LOMO AUC 0.737; swapping Qwen for JetBrains Mellum-4B-sft-python lifts it to 0.764, with a paired bootstrap on `instance_id` putting the delta at +0.031 [+0.016, +0.046] (excludes 0). A 30-line `(file, +/-, payload)` indentation-aware Jaccard baseline ties Mellum at 0.758 (paired delta +0.004 [-0.021, +0.029]); whatever these mean-pooled hidden states recover from diff text is mostly the same lexical overlap a hash-set comparison gives you for free. The post-hoc version (uses the final patch) and the prefix-conditional version (within-model GroupKFold(instance_id) on first-k-turns features holds 0.58-0.60 across k=5..30) together sit in the gap between the authors' predictability and consistency dimensions and would be a natural fifth sub-metric for trajectory-level reliability work.
